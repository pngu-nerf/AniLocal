## What this is
AniLocal is a **light, offline-first, distributable** desktop anime library player. It scans the user's anime folders, identifies files by parsing their names, enriches them with metadata from **AniList** (public API, no account), caches everything locally, and plays via **libmpv (media_kit)**. Built to be downloaded by anyone â€” no server, no account, point at a folder and go. macOS first; Linux/Windows later by recompile.

**Immediate library population (v10):** the library reflects what's on disk the moment it's scanned â€” AniList is enrichment, not a gate for existence. There are now **THREE** file states, not two: **matched** (`file_cache.anilistId` set), **pending** (`anilistId` null + `pendingIdentification` true), and **confirmed-unmatched** (`anilistId` null + `pendingIdentification` false). PENDING = discovered on disk + parsed but not yet identified (offline, not-yet-tried, or a transient lookup failure) â€” shown in the grid as a **named placeholder** (parsed title, blank art) and **retried on every scan** until it resolves; CONFIRMED-UNMATCHED = AniList was consulted and genuinely found nothing (or no parseable title) â€” goes to the fix-match "unmatched" screen, never retried automatically. The scan is **two-phase**: phase 1 writes newly-seen titled files as pending rows with NO network and fires `LibrarySync.sync`'s `onDiscovered` callback (the UI reloads â†’ placeholders paint instantly, even fully offline); phase 2 identifies and **upgrades the same record in place** (a match clears the pending flag and fills title+art; a no-match flips it to confirmed-unmatched; a lookup error LEAVES it pending). A failed/absent AniList never drops a file â€” at worst it stays a named placeholder. Placeholder grouping + the synthetic NEGATIVE series id (`placeholderSeriesId` in `lib/data/cache/placeholder_identity.dart`, derived from the normalized parsed title; can't collide with positive AniList ids) live entirely in the **data layer** (`DriftLibraryRepository`); the UI consumes `Series.pending`. The synthetic id is an **ephemeral display/grouping handle â€” never durably stored**: the one path that could persist it is `watch_state` (you watched a placeholder before it identified), and `applySync`'s `promotions` **rekey those rows from the synthetic id to the real AniList id** the instant the file identifies, so resume progress carries over to the real series and no synthetic id survives identification (covered by a test). Source pins refuse a pending episode outright (`selectSource`/`clearSource` no-op on a negative id; the detail UI hides the picker), so the synthetic id never reaches `source_overrides` â€” seam #5 keeps its no-fill-path-writer guarantee. Match overrides are fingerprint-keyed and store the real target id. An override (fix-match) always makes a file matched, never pending. The v9â†’v10 migration is an additive column defaulting false, so every existing unmatched row keeps its confirmed-unmatched meaning and matched rows are untouched.

**Stable volume identity (v9):** `file_cache` is keyed by `(folder_path, relative_path)` â€” the owning folder's stable identity + the file's path within it â€” NOT an absolute mount path. `library_folders` is bound to a **volume UUID** (`volume_id` + `volume_subpath`) so a removable/NAS volume that remounts under a different `/Volumes` name (collision â†’ `â€¦ 1`, rename, SMB-vs-USB) is **re-found by UUID and doesn't churn** the cache (no re-identify, no AniList refetch). Volume UUIDs **backfill on scan** (needs a mounted volume), so the v8â†’v9 migration only rebases paths by string (identity preserved â†’ first post-migration rescan sees everything unchanged). Fix-match is **fingerprint-based** (stats the present file for `(size, mtime)`) â€” no path scheme, unaffected by remounts.

## Architecture â€” the seams (YOU MUST keep these)
1. **YOU MUST NOT import AniList, Drift, or scanner types inside `lib/ui`.** UI talks to repository interfaces and domain models only.
2. **The cache is the primary read path.** UI reads from cache; the pipeline fills it from AniList at scan/refresh time. The UI MUST never wait on the network. Online vs offline is invisible to the UI.
3. **All AniList access lives in `lib/data/anilist` only.** A schema/API change touches exactly one module.
4. **Identification lives behind one interface** in `lib/data/scanner`. The parser is swappable without touching anything else.
5. **YOU MUST NOT let a rescan overwrite a manual override** â€” match (which AniList entry/episode a file is) OR source (which copy a multi-source episode plays). Both override stores have NO write path from the fill path (`applySync`). User corrections are sacred.

Folders: `lib/ui`, `lib/domain` (models + repository *interfaces*), `lib/data/cache`, `lib/data/anilist`, `lib/data/scanner`, `lib/sync` (pipeline), `lib/playback`.

## Pipeline & cache rules
- Read path: UI -> repository -> cache. Always.
- Fill path (scan/refresh only): scanner -> identifier -> AniList fetch -> write cache. Never on the read path.
- **Incremental only:** rescans process new / moved / removed files. NEVER refetch unchanged items.
- The cache is a **projection** â€” only fields the UI renders, keyed by AniList ID. New field = deliberate Drift migration, not a reflex.

## Single-source-of-truth rules (prevent duplication debt) â€” see `docs/tech-debt-audit.md`
The test for every change: *"to change X, how many places must I edit?"* â€” the answer must be **one**.
- **Reuse before you build.** Before creating a UI element/widget, search for an existing one; if it exists, make it **configurable per location** (flags/slots) and reuse it â€” NEVER build a second parallel version. Two impls of the same thing WILL drift.
- **One source of truth for any value or rule used in 2+ places.** A display value, fallback, or format that appears more than once becomes ONE getter/util (e.g. `Series.displayTitle`, `Episode.displayTitle`, a shared `formatDuration`). Don't inline it again.
- **Cross-cutting config is injected as ONE object, not threaded.** App-wide values (settings) come from a single injected service/repository at the composition root â€” never a fan of individual `load*/set*` functions passed screen-to-screen, and never the same actions bundle constructed in two screens.
- **Never key list rendering or identity by positional index.** A row carries its own item; actions resolve from the row, not `list[index]`. Map selections by identity/value, not by an index into a list that could be filtered/reordered. (This caused a shipped bug.)
- **Prefer live reads over snapshots** for anything that can change while a screen is open. A deliberate snapshot MUST carry a comment saying why it can't go stale.
- **A heuristic that assumes runtime behavior states its assumption at the call site** (stream cadence, event ordering, timing â€” e.g. "a jump > 2s is a seek"), so an unrelated change can't silently break it.
- **Delete code a migration orphans** in the same change â€” no leftover stubs/dead helpers.
- **Do NOT refactor the documented fragile player machinery** (`docs/player-regression-checklist.md`; audit Â§F: non-subscribing fullscreen read, tooltip-dismiss observer, `Listener`-not-`MouseRegion` cursor wiring, fullscreen guards, focus ownership, layout clamps) without first reproducing the bug it fixes.

**Multi-source episodes** â€” âœ… BUILT (schema v7). Library folders are an **ordered priority list** (`library_folders.sortOrder`, top = preferred); the default source is the highest-priority folder that has the episode, falling back down. A per-episode manual source override (`source_overrides`, keyed by episode identity) beats the default and is **sacred across rescans** (seam #5, source dimension â€” no fill-path writer).
